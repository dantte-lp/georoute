# Архитектура

> Control-plane и data-plane делают разную работу. `georoute` синхронизирует
> их из одного источника — странового фида префиксов — не сцепляя одно
> с другим.

## Проблема

Когда вы разделяете exit-ноды по стране назначения, вам нужны *два*
согласованных списка в двух местах:

1. **Control-plane**: BGP должен анонсировать страновые префиксы пирам,
   чтобы трафик с других нод вашей сети «заворачивался» на этот узел.
2. **Data-plane**: локальная FIB ядра должна форвардить совпавший трафик
   в правильный uplink — *без* того, чтобы анонсируемые префиксы попадали
   в main routing table (это бы засорило каждый lookup).

Наивный подход — `redistribute static` + 10 000 blackhole-маршрутов — связывает
эти две вещи и распухает main FIB. `georoute` держит их строго разнесёнными.

## Два плана

```
                       ┌─────────────────────────────┐
                       │  RIPE Stat country list     │
                       │  (HTTPS, идемпотентный fetch)│
                       └──────────────┬──────────────┘
                                      ▼
                       ┌─────────────────────────────┐
                       │         georoute            │
                       │ aggregate → diff → apply    │
                       └──────────────┬──────────────┘
                            ┌─────────┴──────────┐
                            ▼                    ▼
              ┌──────────────────┐    ┌────────────────────┐
              │ /etc/frr/frr.conf │    │  nft inet pbr      │
              │  network X/Y     │    │  set ru_v4/ru_v6   │
              │  (между маркерами)│    │  (атомарный replace)│
              └──────────┬───────┘    └──────────┬─────────┘
                         ▼                       ▼
                ┌─────────────────┐    ┌───────────────────────┐
                │ frr-reload.py   │    │ mark + ip rule fwmark │
                │ → BGP advert    │    │ → lookup table 100    │
                │   пирам         │    │ → out локальный uplink│
                └─────────────────┘    └───────────────────────┘
                CONTROL PLANE              DATA PLANE
```

Эти два выхода независимы на проводе, но логически согласованы потому,
что пришли из одного прохода агрегации.

## Порядок обновления

В рамках одного запуска `georoute` применяет в таком порядке:

1. **Сначала nft** (data-plane). Если новый префикс попадёт в BGP RIB
   раньше, чем nft о нём узнает — пир потенциально притянет трафик
   на dst, которое мы пока не маркируем; пропущенный mark уходит в
   BGP-default, а на транзитной ноде это означает «обратно к пиру». Это
   петля. Обновление nft первым закрывает это окно.

2. **Потом FRR** (control-plane). После `splice → frr-reload.py --reload`
   пиры видят новые префиксы. По построению, наш nft о них уже знает.

Обратный порядок небезопасен; не меняйте его, не подумав про это окно.

## Почему не просто статические маршруты

`redistribute static` с тысячами `ip route ... blackhole` *работает*, но:

- Засоряет main FIB. Каждый сервис на узле теперь продирается через
  10 000 лишних маршрутов (хуже cache-footprint, медленнее `ip route get`).
- Сцепляет «что узнает пир» с «что должно сделать ядро». Два разных режима
  отказа на один список префиксов.
- Не композируется с policy routing: нельзя сказать «VPN-клиенты ходят
  через страновой список, мой собственный SSH-трафик — мимо». С nft+fwmark
  можно.

Interval-set в nft индексирован деревом: O(log n) на 10 000 элементах
неотличим от O(log n) на 100. Память — ~256 КБ.

## Почему `type route` для chain output

```nft
chain output {
    type route hook output priority mangle;
    ...
    ip daddr @ru_v4 meta mark set 0x201
}
```

`type route` говорит ядру: *если* эта цепочка меняет метаданные, релевантные
маршрутизации (mark, source, ToS) — пересчитай routing decision. Без него
локально-порождённый пакет сохраняет решение, принятое *до* выполнения
output-цепочки, и mark, который мы только что установили, игнорируется.
С ним ядро ре-эвалюирует FIB, и наше `ip rule fwmark 0x201 lookup 100`
вступает в силу.

## VRF-гоча: двойной проход netfilter

Когда входной интерфейс — slave VRF (например, ocserv `vpns0` enslave'нутый
в `vrf-vpn`), netfilter видит пакет *дважды*: один раз с `iif=vpns0`, второй
с `iif=vrf-vpn`. Без защиты мы пометим пакет дважды — это в основном
безобидно, но тратит работу. В поставляемом `pbr.nft` в начале цепочки стоит

```
meta iifkind "vrf" return
```

— чтобы не обрабатывать второй проход с VRF-master.

## Что НЕ в scope

- **Поддержка странового списка.** `georoute` тянет из RIPE Stat. Хотите
  другой источник (BGP RIS, RIR delegated stats, antifilter.download,
  собственный курируемый AS-SET) — форкаете fetcher.
- **NAT.** Masquerade для клиентского трафика делается firewalld'ом
  (zone-level) или чем у вас. `georoute` занимается только маршрутизацией,
  не трансляцией адресов.
- **Cross-VRF nexthop.** Если у вас Linux VRF, маршрут в таблице 100 должен
  резолвиться в той же VRF, или используйте синтаксис `nexthop vrf`. Это
  деталь развёртывания; `georoute` ей не управляет.
- **Несколько стран на одну ноду.** Сегодня `RU` против RIPE Stat
  захардкожено. Multi-country (или другой фид) запланированы — см.
  roadmap.

## Roadmap

- Флаги `--country` / `--source`, чтобы фид был подключаемым per-node.
- `--peer-escape`, чтобы исключить точные эндпоинты пиров из сета (чтобы
  случайно не завернуть WG underlay сам в себя).
- Опция BGP large-community для операторов с AS-планами, где это нужно
  (RFC 8092).
- Чистый netlink для nft + vtysh-backed mode для FRR, чтобы убрать
  зависимость от скрипта `frr-reload.py`.
