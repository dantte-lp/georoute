# Гайд для контрибьюторов

Это приватный внутренний инфра-проект, так что круг участников небольшой —
но планка для изменений как у любого продакшена, который трогает BGP и
таблицы маршрутизации ядра.

## Базовые правила

1. **Атомарность важна.** Любое изменение логики применения фида обязано
   держать оба обновления (nftables-сет и FRR splice) в правильном порядке
   и с возможностью отката. Текущий порядок — сначала nft, потом FRR;
   обоснование в [docs/ARCHITECTURE.ru.md](docs/ARCHITECTURE.ru.md#порядок-обновления).
2. **Ноль замечаний линтера.** Запустите `make lint`. Конфиг golangci-lint v2,
   `default: all`. Новый код должен пройти без `//nolint`-директив — за
   исключением случаев, когда подавление пояснено в комментарии той же
   строки.
3. **Однофайловая структура — пока возможна.** Не плодите пакеты, пока файл
   не перевалит за ~600 строк или пока новая логическая подсистема (например,
   второй источник фида) не оправдает собственный пакет.
4. **Никаких внешних рантайм-зависимостей.** Только stdlib. Тесты могут
   использовать `testing`; бенчмарки — `testing.B`. И всё.
5. **Никакой рефлексии, никакого codegen.** Это утилита на ~300 строк,
   держим её скучной.

## Цикл разработки

```bash
make fmt        # gofmt + goimports
make lint       # golangci-lint run
make build      # собирает ./georoute
make test       # unit-тесты
make test-race  # с -race
```

Перед открытием PR прогоните полную цепочку:

```bash
make ci
```

`make ci` — это ровно то, что запускает CI.

## Сообщения коммитов

Conventional Commits, scope-префикс по области:

```
feat(feed): allow country code via --country flag
fix(nft): handle empty v6 prefix set without leaving stale elements
docs(arch): clarify why we mark in output chain with type=route
```

Разрешённые типы: `feat`, `fix`, `refactor`, `perf`, `docs`, `test`,
`chore`, `ci`, `build`.

## Процесс pull request'а

1. Бранч от `main`. Имя бранча: `topic/<short-slug>`.
2. Откройте draft-PR пораньше. CI должен быть зелёным перед review.
3. Минимум один ревьюер. Ревьюеру стоит прогнать новый код в dry-run против
   реального `frr.conf` и убедиться что диф — то, что было задумано.
4. Squash-merge с одним Conventional-Commit-заголовком. Тело merge-коммита
   должно ссылаться на сопутствующие инфраструктурные изменения
   (конфиги на узлах и т.п.), чтобы операционный контекст восстанавливался
   из `git log`.

## Баги и идеи

- Баги: используйте шаблон bug report; приложите точное содержимое блока
  `frr.conf` между маркерами (без секретов) и соответствующий вывод
  `nft list table inet pbr`.
- Фичи: используйте шаблон feature request; обоснуйте предложение против
  «базовых правил» выше.
